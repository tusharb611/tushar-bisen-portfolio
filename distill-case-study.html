The Problem I Kept Running Into

Every project started with a wall of text

At FactSet, new design projects arrived as Confluence pages linked from Jira tickets. These documents contained everything — background context, business requirements, user research data, current UI screenshots, acceptance criteria, success metrics, stakeholder notes — often spanning 8 to 15 pages.

Before I could design a single frame, I needed to distill all of that into what actually mattered: what's the problem, who has it, what does success look like, and what constraints am I working within?

This distillation process took 30–45 minutes every single time — and I was doing it for every project, every sprint, every redesign request. It was high-effort, low-creativity work that ate directly into design time. I suspected my colleagues felt the same way.

So I built a tool to do it for me.

Design Decisions

Why a Figma plugin, specifically

I considered building this as a standalone web tool or a Confluence macro. But a Figma plugin made the most sense for one reason: designers are already in Figma when they need the brief. The brief should live where the design work begins — not in a separate tab they have to context-switch to.

"How might I give designers a structured, scannable brief at the start of every project — without making them leave Figma or read a 10-page document?"

The output format was also a deliberate choice. Rather than generating a PDF or a new document, the brief drops directly into the Figma canvas as formatted text — ready to sit alongside early sketches, be referenced during design reviews, and evolve as the project develops.

How I Built It

Designing and building solo

01

Define the output structure

I started by mapping what a perfect design brief contains — Goal, Problem Statement, User Problems, Required Outcome, Constraints, Success Metrics, Relevant Links. This became both the AI prompt structure and the output template.

02

Prompt engineering

Used Claude and ChatGPT to develop and refine the extraction prompt. The key challenge: getting consistent, hierarchically structured output from highly variable input documents. Tested across 12+ different Confluence docs before the output was reliable enough to use.

03

Build the plugin UI

Designed and built the plugin interface using Claude Code — an upload/paste input area, a project type selector for context, and a generate button. Kept the UI intentionally minimal so it felt native to Figma's own aesthetic.

04

Canvas output rendering

Engineered the output to render directly onto the Figma canvas as formatted text frames — hierarchically structured, with clear section labels and consistent typography so the brief was immediately readable and referenceable.

05

Test with real users

Two colleagues tested the plugin on their own live project briefs. Collected feedback on output accuracy, missing sections, and formatting issues. Iterated on the prompt and output structure based on their findings before settling on the final version.

🎯 Goal
Redesign the Screening Assistant criteria flow to support Generative AI-powered query input

Enable analysts to define screening criteria using natural language prompts via the integrated Mercury AI chat — reducing manual configuration overhead and expanding accessibility to non-technical users.

🔍 Problem Statement

The existing Screening Assistant requires analysts to build criteria logic manually through a structured form interface. This limits speed, creates a steep learning curve for new users, and does not leverage AI capabilities now available in the platform.

👤 User Problems

• Manual criteria building is slow for complex, multi-condition screens

• New analysts struggle to learn the criteria logic structure without training

• Users cannot easily translate natural language investment theses into screening logic

• No way to preview or validate criteria intent before running a screen

✓ Required Outcome

• AI chat input that translates natural language into screening criteria

• Criteria preview with plain-language explanation before running

• Ability to edit AI-generated criteria manually after generation

• Graceful fallback when AI cannot interpret the query

• ↓ Time to first screen run for new users

• ↓ Support tickets related to criteria configuration

Reflection

What I learned building this

This project taught me something I hadn't fully appreciated before: the best tools solve for the space between other tools. Confluence and Figma both work well independently — the gap was in moving from one to the other. Distill lives in that gap.

What worked well

  • Solving my own pain first made the requirements crystal clear — I didn't need a research plan to know what the tool should do
  • Using Claude Code to build the plugin let me focus on design decisions rather than implementation syntax
  • Testing with real colleagues on real project documents exposed edge cases that sanitised test data never would have
  • Confidence indicators (flagging ambiguous or missing sections) significantly improved trust in the output

What I'd build next

  • FigJam export — drop the brief as a sticky note cluster on a board for easier workshop use
  • Follow-up chat — let designers ask the AI to clarify or expand on any section of the brief
  • Template library — different brief formats for New Feature vs Redesign vs Research projects
  • Team sync — share briefs across a Figma org so the whole team works from the same starting point
