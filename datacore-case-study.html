Overview

A legacy system in need of a new foundation

DataCore's SANsymphony (SSV) platform is a software-defined storage solution used by IT engineers and storage administrators to manage complex server group configurations, disk pools, and virtual storage environments.

The existing platform was a Java desktop application — technically functional but visually dated, cognitively heavy, and increasingly misaligned with how modern IT teams work. DataCore wanted to migrate to a web-based console while simultaneously redesigning the experience from the ground up.

The biggest hidden problem: the platform was entirely reactive. Users only found out something was wrong when a crisis had already arrived. There was no proactive monitoring, no predictive insights, and no way to get ahead of issues before they impacted operations.

Research

What the audit and interviews revealed

🔬

UI audit

A systematic evaluation of the Java platform revealed deep inconsistencies in navigation patterns, redundant workflows, and excessive information density that obscured critical signals.

👨‍💻

Engineer interviews

Client-side engineers and storage admins walked us through their real workflows — revealing workarounds, missed alerts, and configuration routines that the existing UI made unnecessarily complex.

⚠

Reactive monitoring gap

The most critical finding: users only discovered storage issues after they became operational problems. There was no mechanism to surface anomalies before they escalated.

🏗

Scalability challenge

The Java interface had no scalable design framework — each module had evolved independently, creating a patchwork of patterns that made cross-module consistency impossible.

Problem Framing

Three design challenges

"How might we migrate DataCore's storage management experience from a dense Java application to a scalable web console — while shifting users from reactive firefighting to proactive system intelligence?"

1. Java → Web Migration

Map existing SSV flows to modern, task-focused web views without losing the depth that power users depend on.

2. Scalable Framework

Build a modular design system that supports all DataCore modules consistently and allows flexible configuration as the product grows.

3. Proactive Intelligence

Design DataCore Insight Services — a new module that surfaces system health, disk behaviour, and performance anomalies before they become problems.

Outcomes

Impact

📈

+53% platform adoption

Within the first quarter of the web console rollout, over half of DataCore's user base had migrated to the new platform — a testament to the quality of the transition experience.

⚙

↓30% configuration time

The streamlined task-focused web views and consistent IA significantly reduced the time users spent navigating and configuring their storage environments.

🔮

Proactive vs reactive

DataCore Insight Services fundamentally shifted how users interacted with system health — from discovering problems after the fact to preventing them with predictive guidance.

🏗

Scalable design framework

The modular design system established in this project became the foundation for future DataCore modules — significantly reducing design and engineering overhead on subsequent features.

Reflection

Key learnings

What worked well

  • Starting with a thorough UI audit gave us an objective, evidence-based foundation for redesign decisions — not just opinions
  • Designing the Insight Services module as a separate layer (not embedded in existing views) gave it room to evolve without disrupting core workflows
  • Involving engineers early in the design process ensured technical feasibility and faster handoff
  • Grooming junior designers throughout kept quality high across a large, parallel workstream

What I'd do differently

  • Establish a formal onboarding flow — the migration from Java to web console was smooth for power users but required more support for less technical admins
  • Design feedback loops into Insight Services so the system learns which insights users find most valuable over time
  • Document the design system more thoroughly during the project rather than after — it would have reduced the ramp-up time for new team members
